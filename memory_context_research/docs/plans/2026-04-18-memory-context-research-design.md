# memory_context_research 设计

Date: 2026-04-18
Status: approved for implementation

## 目标

构建一个独立运行的每日研究工作流，用来：

- 跟踪一组上下文 / 记忆相关的代码仓
- 每天分析这些代码仓的增量变化
- 识别真正有意义的新产品能力，而不是简单罗列代码改动
- 评估这些新能力是否可能对 OpenViking 产生帮助
- 将结果写成仓库内的一份精简 Markdown 日报

第一版优先服务于人的阅读和判断，不以 Agent 直接调用为第一目标。

## 产品要求

第一版确认的边界如下：

- 以独立后台任务的形式每天定时运行
- 以代码 diff 作为主信号源
- 以 PR、changelog、release note 作为辅助证据
- 以仓库内 Markdown 报告作为主要输出形式
- 结论层以产品视角为主
- 技术细节只保留足以支撑结论的部分

## 范围

第一版包含：

- 研究对象代码仓清单配置
- 每日定时调度执行
- 本地代码仓镜像更新 / 拉取
- 基于上次成功游标的增量分析
- 与上下文 / 记忆系统相关的变更过滤
- 精简日报生成
- 用于追溯和调优的中间 artifacts 落盘

第一版不包含：

- plugin 或 MCP 接口
- 面向聊天交互的主流程
- Slack / 飞书 / 邮件等消息推送
- 复杂 Web UI
- 强依赖写回 OpenViking 记忆存储
- 自动创建 issue 或 roadmap 建议

## 推荐形态

主体应实现为独立研究服务，而不是直接做成 skill 或 plugin。

### 为什么不优先做 Skill

skill 更适合表达一种可复用的 Agent 能力，比如“分析代码仓演进”或“横向比较最近的记忆系统变化”。但它不适合作为这套系统的一期主体，因为这里还有很多后台职责：

- 定时调度
- 仓库镜像管理
- 增量分析游标持久化
- 多代码仓批处理
- 失败隔离与重试
- 日报文件生命周期管理

因此，skill 更适合作为第二阶段的结果消费入口，而不是承担每日执行任务的引擎。

### 为什么不优先做 Plugin

plugin 或 MCP 服务更适合“外部宿主交互式查询结果”的场景。当前需求本质上是一个内部研究流水线。若一开始先做 plugin，会过早引入接口设计、宿主集成和鉴权等额外工作。

### 推荐演进路径

第一阶段：

- 独立的每日研究流水线
- 仓库内 Markdown 日报
- 中间结果持久化

第二阶段：

- 增加一个 skill，用来读取日报 / artifacts 并回答更高层的比较问题

第三阶段：

- 当交互式查询成为明确需求后，再通过 plugin 或 MCP 接口对外暴露能力

## 架构

系统建议拆成五层。

### 1. Source Registry

负责维护研究对象代码仓及其分析偏好。

每个仓库配置建议包含：

- `name`
- `repo_url`
- `default_branch`
- `category`
- `watch_paths`
- `signals`
- `notes`

### 2. Fetcher

负责更新本地镜像，并根据上次成功游标计算增量范围。

职责包括：

- 对未存在的代码仓执行 clone
- 对已存在的代码仓执行 fetch
- 解析 `last_commit -> HEAD` 的增量区间
- 对首次接入的仓库限制初始化分析窗口

### 3. Change Analyzer

以 diff 为主输入，抽取值得深入研究的候选变更。

职责包括：

- 过滤明显噪音，如纯格式化、依赖升级、CI 调整和无关文档修订
- 优先关注 `watch_paths` 覆盖目录和与记忆 / 上下文能力相关的文件
- 摘要出模块、接口、schema、prompt、retrieval 逻辑等方面的变化

### 4. Insight Distiller

将技术层证据压缩成产品层结论。

输出内容包括：

- 新增了什么能力
- 这个能力大概率在解决什么问题
- 它属于哪一类上下文 / 记忆系统演进
- 它是否可能对 OpenViking 有帮助
- 对应的证据和置信度

### 5. Report Writer

负责写出最终 Markdown 日报，并保存原始 / 中间分析结果。

## 数据模型

第一版采用轻量、文件化的数据模型。

### `config/repos.yaml`

存放研究对象清单和分析提示信息。

### `config/state.json`

保存每个仓库的增量游标和运行状态。

### `research/artifacts/YYYY-MM-DD/<repo>/`

保存每天、每个仓库的中间产物。

### `research/daily/YYYY-MM-DD.md`

保存最终给人阅读的日报。

## 日报格式

日报应优先服务于快速扫读和决策。

每个仓库的小节需要回答：

- 今天有哪些变化值得关注
- 为什么这些变化重要
- 对 OpenViking 可能有什么帮助
- 这些判断的依据是什么

建议格式：

```md
## repo-a
结论：新增了一项长期记忆召回相关能力。

新增功能：
- 引入 xxx
- 支持 xxx

为什么重要：
- 解决 xxx 问题
- 说明产品从 xxx 转向 xxx

对 OV 的潜在价值：
- 可能适用于 retrieval / session / schema 层
- 优先级：高

证据：
- commits: ...
- PRs: ...
- changelog: ...
```

## 执行流程

每日任务固定成八步：

1. 读取 `config/repos.yaml`
2. 更新或拉取每个目标代码仓的本地镜像
3. 根据 `config/state.json` 解析本次增量范围
4. 基于 git diff 提取候选变更
5. 将候选变更分析为能力级结论
6. 用 PR、changelog、release note 对结论交叉取证
7. 写出 Markdown 日报和对应 artifacts
8. 仅在该仓库分析成功后才更新它的游标状态

## 失败处理

第一版优先保证结论可信、日报可读。

- 单个仓库分析失败，不应导致整次日报任务失败
- 某仓库只有在成功完成分析并生成结果后，才允许推进游标
- 每个仓库都应保存原始证据，便于复查
- 噪音过滤应偏保守，避免日报被低价值变更淹没
- 每条结论建议附带简单置信度，如 `high`、`medium`、`low`

## MVP

最小可用版本包含：

- 文件化的代码仓清单配置
- 每日定时执行
- 基于 git diff 的增量分析
- PR / changelog / release note 辅助归因
- artifacts 落盘
- 单份 Markdown 日报输出

MVP 不建议包含：

- plugin 封装
- Web UI
- 消息推送
- 自动建 issue
- 将 OpenViking 写回作为强依赖
